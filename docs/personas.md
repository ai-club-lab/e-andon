# ペルソナ別ユーザーフロー

チョコ停1件のライフサイクル **検知 → 通知 → 対応宣言 → 対処 → 裁定 → 学習** を、
関わる人ごとに「きっかけ / 見るもの / すること / システムの応答」で規定する。
当番名簿はデモ用の架空名簿（routing_rules テーブル。実運用は UPDATE で差し替え）。

> **デモの配役（Slack 2アカウント）**: メインアカウント（aiclubhackathon）＝一次対応者
> 「保全・安藤さん（搬送担当）」、サブアカウント＝「班長・鈴木さん」。`SLACK_PERSONAS`
> （ユーザーID→名前・`;`区切り）で個別に割当て、未登録IDは `SLACK_PERSONA_DEFAULT`（安藤さん）に落ちる。

## 1. 保全・安藤さん（一次対応者 — 搬送担当）

主面: **Slack + スマホ**。ラインの近くにいるとは限らない。手袋・立ち作業。

| フェーズ | 安藤さんの体験 |
|---|---|
| 通知 | Slackカード着信:「⚠ 整列異常でライン停止。真因は『◯◯』（確信度・根拠・前回の対処・現場写真つき）。👥 担当: **保全・安藤さん** に通知しました」 |
| 対応宣言 | カードの **👋 私が対応します** を押す → スレッドに「👋 保全・安藤さんが対応します — 以降の自動連絡は止めました」。**班長への5分後連絡が止まる**。Web/モバイルにも数秒で同期 |
| 対処 | deep link でスマホ裁定ページへ。AIの根拠・類似過去事例・前回の処置・現場写真を見て初動を短縮 |
| 裁定 | 直したら **✅ 正しかった / ❌ 原因が違った**。❌ならAIが対話で聞き取り（スレッド or チャット）、現場写真を1枚添付可能 → 事例化されて次回の推論が変わる |
| 例外 | スレッドに雑談・質問・対応表明を書いても**訂正扱いにはならない**（訂正対話は❌で明示的に開く）。質問ならAIがログを調べて答える |

## 2. 班長・鈴木さん（エスカレーション先 / 二次裁定者）

主面: **Slack**。持ち場は別。呼ばれた時だけ動く。

- **きっかけ**: 安藤さんが5分無応答 →「5分たっても応答がないため、**班長・鈴木さん** に連絡します。対応できる方はカードの『👋 私が対応します』を押してください」
- **すること**: 自分が向かうなら👋（実名で全面に共有される）。現場を知る立場として、AIの推定への裁定も遠隔から可能（**裁定は対応者でなくてもよい** — 推論の正誤と対処の担当は独立）
- **さらに無応答**: +15分でベンダー連絡先が提示される（自動発報はしない — 対外連絡は人の判断）

## 3. 監視オペレーター（Web常駐 — 中央監視室）

主面: **Webダッシュボード**。全ライン俯瞰・映像・センサー・AIチャット。

- **平常時**: 稼働状況・波形を監視。AIチャットでログ照会（「各センサーは正常？」）
- **停止時**: カードが着信 → 誰に通知されたか・誰が対応中か・裁定の結果が**現場と同じ内容・同じ順序**でリアルタイムに見える（4秒ポーリング）。自分が対応する場合は👋から当番チップ or 名前入力
- **価値**: 現場に電話して「誰かやってる？」と聞く必要がなくなる（状態が画面にある）

## 4. 製造課長・伊藤さん（管理者 — 週次改善）

主面: **/analytics**。tier3の最終連絡先でもある。

- **週次**: 原因パレート（どの真因で何分止まったか）・AI正答率の推移・再発間隔を見て、恒久対策（治具交換・点検周期）の優先度を決める
- **価値**: チョコ停が「なんとなく多い」から「◯◯起因が今週N件・計M分」になる

## 5. 審査員（デモ閲覧者）

- 稼働URLを開くだけで1サイクルを体験（ガイドが面の役割を説明: 通知の本番面はSlack、Webは監視・分析用。デモでは同じ操作をWebでも体験可能）
- 触ってよい: 👋（当番チップ）・裁定・チャット・分析。すべて本物のループ（裁定は正答率と事例に反映される）

## サイドバーの情報設計（誰が・いつ・どの頻度で使うか）

| グループ | 中身 | 主な使い手と頻度 |
|---|---|---|
| **監視 — 生産ライン** | ライン一覧（LIVE/デモ・状態ドット） | 監視オペが**常時**。停止の一報と映像・センサーの確認 |
| **分析・改善** | 📊 分析・履歴（再発・パレート・正答率・学習データの由来） | 課長・班長が**週次/朝会**。恒久対策の優先度決め |
| **運用・管理** | 👥 当番表（カテゴリ→通知先→無応答時の連絡順）・⚙ 設定 | 班長が**シフト替わり/配置換え時**。「誰に飛ぶか」の確認と変更 |

頻度の高いものほど上。監視は「いま」、分析は「ふりかえり」、管理は「しくみ」— 時間軸で分けている。

## フロー順序の設計判断

- **👋（対応宣言）→ 対処 → 裁定** の順。カードのボタンもこの順に並ぶ。裁定ボタンには「対応が終わったら」のキャプション
- **裁定に対応宣言を必須にしない**: 推論の正誤の判断（知識）と現場対処（体）は別人でありうる（例: 鈴木さんが遠隔で裁定、安藤さんが対処）
- **先着1名 first-wins**: 2人目の👋には「先に◯◯さんが対応中です」。裁定も同様（後勝ちしない・監査ログに残る）
- **訂正対話は明示的に開く**: ❌ボタンだけが訂正の入口。スレッド発言を勝手に訂正と解釈しない（決定論ゲート）
- **未実装（決勝スペック）**: ✅復旧しました（停止時間の実測クローズ）・シフト連動の当番表・通知ストーム集約
