# Requirements — andon-human-loop

> ステータス: **requirements-generated**
> 記法: EARS（Ubiquitous / Event-driven `WHEN` / State-driven `WHILE` / Unwanted `IF..THEN` / Optional `WHERE`）
> 主語: 「本システム」= e-Andon（親スペック `chokotei-anomaly-rca` の稼働システムを拡張）
> 注記: `.kiro/steering/` が未整備のため、全体文脈は本要件・親スペック・`docs/audit.md`・`docs/observability.md` を一次ソースとする。

## プロジェクト概要

e-Andon の AI ループ（検知 → 原因推定 → 裁定 → 学習）は稼働済みだが、現状は
「ダッシュボードを見ている人」しか救えない。本フィーチャーは未実装の**人間側ループ**
（通知 → 駆けつけ → 修理 → 確認）を追加し、アンドン本来のプッシュ型・責任者接続を実現する。

## ペルソナ（ユーザーストーリーの主語）

| ペルソナ | 置かれた状況 | 満たすべき瞬間 |
|---|---|---|
| **オペレーター** | ラインの近く。歩き回る。手袋・立ち作業 | 止まった瞬間に「何をすればいいか」をスマホで知る |
| **保全担当** | 別の建屋にいる | 自分が呼ばれるべき停止**だけ**通知される |
| **班長** | 複数ラインを統括 | 対応が長引いたときだけ割り込まれる |
| **工場長／生産技術** | 週次・月次で改善判断 | どの真因で何分失ったか・再発は何か・AIは役に立っているか |

## 確定済み前提（親スペック・意思決定より）

- 重い意思決定（異常確定・停止・通知先決定・エスカレーション発火）は **LLM の外（決定論）＋監査ログ**
- ステートフル・シングルトン構成（`--min-instances=1 --max-instances=1`）を維持
- 通知シンクは**差し替え可能な抽象**とする。デモは Slack（現実の工場は LINE WORKS / Teams / パトランプ等）
- 訂正の学習還流（埋め込み → pgvector → few-shot）は既存の `/feedback` コードパスを再利用する

---

## Requirement 1: 通知シンク抽象と Slack カード通知

**User Story:** オペレーターとして、ダッシュボードを開いていなくても停止と AI の見立てを即座に受け取りたい。アンドンは向こうから知らせに来るものだから。

#### Acceptance Criteria
1. 本システムは、通知シンクを差し替え可能な抽象インターフェースとして定義し、Slack を最初の実装とするものとする。
2. WHEN チョコ停イベントの RCA が完了したとき、本システムは通知先 Slack チャネルへ「ライン名・停止時刻・真因候補・確信度・根拠（ツール呼び出し履歴を含む）・代表フレームへの参照」を含むカードを1回だけ投稿するものとする。
3. 本システムは、通知カードにダッシュボードの該当イベント画面への深いリンクを含めるものとする。
4. IF Slack への配信が失敗した場合、THEN 本システムは無音でフォールバックせず、失敗を構造化ログに記録し、ダッシュボード上に配信失敗を明示するものとする。
5. 本システムは、同一イベントに対する通知の重複投稿を防止するものとする（再起動・リトライ後も同一イベント ID で高々1カード）。

## Requirement 2: Slack 上での HITL 裁定

**User Story:** オペレーターとして、通知カードからその場で「正しい／違う」を裁定したい。ダッシュボードを開きに行く往復をなくすため。

#### Acceptance Criteria
1. 本システムは、通知カード上に「正しい」「違う」の裁定操作を提供するものとする。
2. WHEN ユーザーがカード上で裁定したとき、本システムは既存の HITL 裁定と同一のコードパス・同一のデータストアに記録するものとする。
3. 本システムは、裁定状態をダッシュボードと Slack の両面で単一の真実として扱い、一方で裁定済みのイベントは他方でも裁定済みとして表示するものとする。
4. IF 既に裁定済みのイベントに対して再度裁定操作が行われた場合、THEN 本システムは二重記録せず、裁定済みである旨と裁定者・裁定時刻を提示するものとする。
5. WHEN 裁定が記録されたとき、本システムは通知カードを裁定結果（裁定者・時刻・結果）を反映した表示に更新するものとする。

## Requirement 3: Slack スレッドでの訂正対話と学習還流

**User Story:** オペレーターとして、AI の見立てが違うとき、スレッドへの返信だけで正しい真因を伝えたい。フォーム入力や画面遷移なしで現場の知見を残すため。

#### Acceptance Criteria
1. WHEN ユーザーがカード上で「違う」を選んだとき、本システムは当該カードのスレッドで訂正エージェントとの対話を開始するものとする。
2. 本システムは、スレッド返信を訂正エージェントへの入力として扱い、既存の訂正対話（深掘り質問は1つまで）と同一の振る舞いをするものとする。
3. WHEN 訂正エージェントが真因を確定したとき、本システムは既存の `record_correction` 経路で埋め込み化し past_cases へ還流するものとする（書き込み先イベントはサーバが固定）。
4. WHEN 訂正が確定したとき、本システムはスレッド上に確定した真因を要約して提示するものとする。
5. IF スレッドでの対話が確定に至らないまま一定時間（既定 30 分）経過した場合、THEN 本システムは対話を未確定のままクローズし、ダッシュボードから訂正を継続できる旨をスレッドに提示するものとする。

## Requirement 4: 裁定者アイデンティティと監査記録

**User Story:** 班長として、誰がいつどの裁定をしたかを後から追いたい。品質記録と再発時の聞き取りのため。

#### Acceptance Criteria
1. WHEN Slack 上で裁定・訂正が行われたとき、本システムは操作者の Slack ユーザー ID・表示名・操作時刻を当該イベントの監査記録に含めるものとする。
2. WHEN ダッシュボード上で裁定・訂正が行われたとき、本システムは操作面（dashboard）を識別子として監査記録に含めるものとする。
3. 本システムは、専用のログイン機構・OAuth 連携を導入しないものとする（Slack のアイデンティティを操作者記録として代用する）。
4. 本システムは、裁定・訂正・エスカレーションを含む監査記録を既存の監査ログと同一の永続ストアに保存するものとする。

## Requirement 5: 真因カテゴリ→担当者ルーティング（決定論）

**User Story:** 保全担当として、自分が直せる種類の停止だけ自分宛てに通知されてほしい。全通知を追う負担と見逃しをなくすため。

#### Acceptance Criteria
1. 本システムは、RCA の出力から真因カテゴリ（例: 位置決め機構／搬送ガイド／センサー系／その他）への分類を提供するものとする。
2. 本システムは、真因カテゴリ×担当者（当番表）の対応を決定論テーブルとして保持し、LLM の出力が通知先の決定に直接使われないものとする（LLM の役割はカテゴリ推定の材料提供まで）。
3. WHEN 通知カードを投稿するとき、本システムはルーティングテーブルに基づく一次担当者へのメンションを含めるものとする。
4. IF 真因カテゴリがテーブルに未登録の場合、THEN 本システムは既定の通知先（班長）へルーティングし、未登録カテゴリの発生を記録するものとする。
5. 本システムは、ルーティングテーブルを再デプロイなしに更新可能な形（設定またはデータストア）で保持するものとする。
6. 本システムは、ルーティング決定の根拠（カテゴリ・テーブル版・決定時刻）を監査記録に含めるものとする。

## Requirement 6: 無応答エスカレーション

**User Story:** 班長として、一次担当が動けないときだけ自分に割り込んでほしい。すべての停止に反応する必要はないが、放置は許されないため。

#### Acceptance Criteria
1. WHILE 停止イベントが未裁定である間、本システムは経過時間を追跡するものとする。
2. IF 一次通知から既定時間（既定 5 分）以内に裁定・応答がない場合、THEN 本システムは第2段の通知先（班長）へエスカレーション通知を行うものとする。
3. IF 第2段通知から既定時間（既定 15 分）以内に裁定・応答がない場合、THEN 本システムは第3段の連絡先（設備ベンダー窓口）の情報をスレッドおよびダッシュボードに提示するものとする（外部への自動発報はスコープ外）。
4. WHEN いずれかの段で裁定・応答が行われたとき、本システムは以降のエスカレーションを停止するものとする。
5. 本システムは、エスカレーションの各発火（段・宛先・時刻）を監査記録に含めるものとする。
6. 本システムは、エスカレーションの発火判定を決定論（タイマー×状態）で行い、LLM を関与させないものとする。

## Requirement 7: 分析ビュー（パレート・再発検知・AI 正答率）

**User Story:** 工場長として、どの真因で何分失っているか・同じ問題が繰り返していないか・AI は当たるようになっているかを1画面で把握したい。恒久対策の優先順位を決めるため。

#### Acceptance Criteria
1. 本システムは、サイドメニューの「準備中」項目を分析ビューへの導線に置き換えるものとする。
2. 本システムは、真因カテゴリ別の停止回数と累計停止時間（損失分）をパレート図（件数降順＋累積比率）として表示するものとする。
3. 本システムは、期間（例: 直近7日／30日）を切り替えて集計できるものとする。
4. WHEN 同一真因カテゴリの停止が既定回数（既定: 7日間に3回）以上再発したとき、本システムは分析ビュー上で再発として強調し、恒久対策の検討を促す能動的な指摘を表示するものとする。
5. 本システムは、AI 推定の正答率（裁定に基づく）の推移を時系列グラフとして表示するものとする（既存 `/metrics` の可視化）。
6. 本システムは、分析ビューの各グラフから根拠となるイベント一覧へ辿れるものとする。
7. IF 集計対象データが存在しない期間が選択された場合、THEN 本システムはデータなしを明示するものとする（空グラフを暗黙表示しない）。

## Requirement 8: 裁定・訂正画面のモバイル最適化

**User Story:** オペレーターとして、Slack 通知から開いた画面をスマホ縦持ち・手袋のままで操作したい。現場は立ち作業でデスクがないため。

#### Acceptance Criteria
1. 本システムは、通知カードの深いリンクから開くイベント詳細・裁定・訂正画面をスマホ縦画面（幅 390px 基準）で横スクロールなしに表示するものとする。
2. 本システムは、裁定・訂正の主要操作ボタンをタップ目標 44px 以上で提供するものとする。
3. 本システムは、モバイル表示において裁定に必要な情報（真因候補・確信度・根拠・代表フレーム）を1画面目に収めるものとする。
4. 本システムは、ライン俯瞰ダッシュボード（管理者向け大画面）のレイアウトを本要件の対象外とするものとする（裁定・訂正動線のみ最適化）。

## Requirement 9: 訂正時の現場写真添付（オプション）

**User Story:** オペレーターとして、実際の原因箇所を1枚撮って訂正に添えたい。言葉で説明しにくい現物の状態を次回の推論に活かすため。

#### Acceptance Criteria
1. WHERE 訂正対話において、本システムはユーザーが写真1枚を任意添付できるものとする（Slack スレッドへの画像投稿、またはモバイル訂正画面からの添付）。
2. WHEN 写真付きで訂正が確定したとき、本システムは写真を非公開ストレージに保存し、past_cases の当該事例に参照を紐づけるものとする。
3. WHEN 類似事例として写真付き事例が検索されたとき、本システムは推論の文脈に写真をマルチモーダル入力として含めるものとする。
4. 本システムは、写真の添付をスキップしても訂正が完結するものとする（添付は必須にしない）。
5. IF 添付ファイルが画像でない、または既定サイズ（10MB）を超える場合、THEN 本システムは受け付けず理由を提示するものとする。
6. 本システムは、写真の保存先を公開 URL にしないものとする（署名付き参照等でのみ提示）。

## Requirement 10: 非機能要件（ガードレール・構成・秘密・可観測性）

**User Story:** 運用者・審査員として、拡張後も既存の設計原則（決定論ガードレール・監査・低コスト・鍵レス）が崩れていないことを確認したい。

#### Acceptance Criteria
1. 本システムは、通知先決定・エスカレーション発火・裁定記録の書き込み先を決定論ロジックに置き、LLM の出力がこれらを直接駆動しないものとする。
2. 本システムは、ステートフル・シングルトン構成（min/max-instances=1）を維持し、通知・エスカレーションのタイマー状態を再起動後に永続ストアから復元するものとする。
3. 本システムは、Slack のトークン等の秘密情報をリポジトリに含めず、Secret Manager で管理するものとする。
4. 本システムは、Slack からの着信リクエストの真正性を検証するものとする（署名検証。検証失敗は拒否し記録）。
5. 本システムは、通知・裁定・エスカレーション・写真添付の各処理を既存の構造化ログ（severity / event_id フィルタ可能）に出力するものとする。
6. 本システムは、Slack 連携が無効な環境（ローカル・CI）でも既存機能とテストが Slack なしで動作するものとする。

## スコープ外（P1）

- LINE WORKS / Teams / パトランプ等の追加シンク実装（抽象だけ用意し、実装は Slack のみ）
- 設備ベンダーへの自動発報・部品発注起票（第3段は連絡先提示まで）
- 当番表のシフト管理 UI（テーブルは設定/データとして更新可能なら足りる）
- 常時カメラ録画の学習蓄積（訂正時の1枚添付のみ）
- 多言語対応・複数工場テナント

## 未決事項（Design 以降で確定）

- Slack 着信（ボタン・スレッド返信）の受信方式: Events API + Interactivity（要公開エンドポイント）か Socket Mode か
- エスカレーションタイマーの実装: プロセス内タイマー＋DB 復元 か Cloud Tasks 等の外部スケジューラか（シングルトン前提との整合）
- 真因カテゴリの語彙設計: 既存 RCA 出力からの後付け分類か、RCA プロンプトへのカテゴリ出力の組み込みか
- 分析ビューの描画方式（既存の軽量フロントに合わせる）
- 写真のマルチモーダル還流を検索ヒット時に毎回含めるか、確信度が低いときのみか
