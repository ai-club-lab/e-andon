# Requirements — chokotei-anomaly-rca

> ステータス: **requirements-generated**
> 記法: EARS（Ubiquitous / Event-driven `WHEN` / State-driven `WHILE` / Unwanted `IF..THEN` / Optional `WHERE`）
> 主語: 「本システム」= チョコ停検知・原因推定エージェント（製品名は暫定）
> 注記: `.kiro/steering/` が未整備のため、全体文脈は本要件と `docs/poc/findings.md` を一次ソースとする。

## プロジェクト概要

工場ラインの俯瞰映像から部品の整列異常（等間隔配置からの逸脱＝縦ズレ／回転）を決定論CVで検知し、
異常発生時にエージェント（ADK/Gemini）が振動等のIoTログを相関抽出して原因を推定、
ダッシュボードのチャットでユーザーに通知する。ユーザーは他ログもチャットから照会できる。
人が確認しAI推定と異なる場合は正解データとして蓄積し、次回の few-shot 改善に再利用する（HITL学習ループ）。
GCPサービスを合理的に用い、1閉ループを稼働URLとして成立させる。

## 確定済み前提（意思決定・PoCより）

- 検知は**ハイブリッド二段**（1段目 全フレームCV／2段目 境界帯のみ Gemini Vision 確認）
- エージェント基盤は **ADK**（AgentTool 構成・セッション永続・バージョン固定）
- 映像は**擬似ストリーム**（`factory_01.mov` を約5fpsでSSE配信）
- PoC 実測：正常ラインは cy σ≈1px / 角度 σ≈1°、実異常は「部品1個が約18px上方へ変位」

## Requirement 1: 映像取り込みと擬似ストリーム配信

**User Story:** ライン監視者として、ラインの映像をダッシュボードでリアルタイムに近い形で見たい。異常が今起きていることを体感するため。

#### Acceptance Criteria
1. 本システムは、指定された映像ソース（`factory_01.mov`）を約5fpsにサンプリングして処理するものとする。
2. WHEN 映像の末尾に到達したとき、本システムはループ再生として先頭から処理を継続するものとする。
3. 本システムは、各配信フレームに検知オーバレイ（ベースライン・部品マーカー・異常マーカー）を重畳して配信するものとする。
4. 本システムは、フレームをSSEまたはWebSocketでダッシュボードへ逐次配信するものとする。
5. IF 映像ソースが読み込めない場合、THEN 本システムはエラーを記録し、ダッシュボードに配信停止状態を表示するものとする。

## Requirement 2: 整列異常の検知（ハイブリッド二段）

**User Story:** 品質担当として、等間隔・同一向きの整列から外れた部品を自動で見つけたい。目視監視の見逃しを防ぐため。

#### Acceptance Criteria
1. 本システムは、各フレームでベルトROI内の部品を検出し、重心座標・回転角・隣接間隔を算出するものとする。
2. WHEN 部品の縦オフセットがフレーム内中央値から閾値（既定10px）を超えたとき、本システムは当該部品を異常候補とするものとする。
3. WHEN 部品の回転角がフレーム内中央値から閾値（既定10度）を超えたとき、本システムは当該部品を異常候補とするものとする。
4. WHEN 隣接部品間隔が中央値の1.5倍を超えたとき、本システムは当該位置に変位／欠落があるとみなすものとする。
5. WHERE 異常度が境界帯（既定 縦オフセット8〜12px）にある場合、本システムは当該フレームの切り出し画像を Gemini Vision に送り整列異常の有無を確認するものとする。
6. IF 1フレームで検出部品数が最小値（4）未満の場合、THEN 本システムは当該フレームを判定対象外とし、その旨を記録するものとする。
7. 本システムは、各判定の根拠（数値・シグナル種別）を監査可能な形で記録するものとする。

## Requirement 3: 異常イベントの生成と時系列集約

**User Story:** 運用者として、同一の異常を1件のイベントとして扱いたい。100フレーム分の重複通知を避けるため。

#### Acceptance Criteria
1. 本システムは、連続フレームにわたる同一変位部品を追跡し、単一の異常イベントに集約するものとする。
2. 異常イベントは、イベントID・開始／終了時刻・ピーク逸脱量・シグナル種別・代表フレーム画像参照を含むものとする。
3. WHEN 新規の異常イベントが確定したとき、本システムは一度だけ下流（推定・通知）をトリガーするものとする。
4. IF 異常が閾値未満に戻り一定時間継続した場合、THEN 本システムは当該イベントを終了とみなすものとする。

## Requirement 4: IoTログの生成・保存と相関抽出

**User Story:** 品質担当として、異常時刻前後の振動・温度・電流を確認したい。物理的な原因の裏付けを取るため。

#### Acceptance Criteria
1. 本システムは、映像タイムラインに整合した合成IoTデータ（振動・温度・モータ電流）を生成し保存するものとする。
2. WHEN 異常イベントが発生したとき、本システムは異常時刻を中心とした時間窓のIoTログを抽出できるものとする。
3. 本システムは、異常イベントに相関する物理シグナル（例：X軸加速度スパイク）を含むよう合成データを構成するものとする。
4. 本システムは、IoTログを時刻・チャネルで問い合わせ可能な形で永続化するものとする。

## Requirement 5: 原因推定エージェント

**User Story:** ライン監視者として、異常の推定原因を根拠付きで知りたい。次のアクションを判断するため。

#### Acceptance Criteria
1. WHEN 異常イベントがトリガーされたとき、本システムは異常内容と相関IoTログをエージェントに渡し原因を推定するものとする。
2. 本システムは、エージェントのツール呼び出し（IoTログ照会・過去事例検索・フレーム画像取得）を AgentTool 構成で提供するものとする。
3. 本システムは、推定結果に原因候補・確信度・根拠（参照した数値／ログ）を含めるものとする。
4. 本システムは、過去の確定事例（RAG）を検索し推定の few-shot 文脈として利用するものとする。
5. 本システムは、エージェントのセッション状態を永続化し、インスタンス間で履歴を失わないものとする。
6. IF 推定モデル呼び出しが失敗した場合、THEN 本システムは無音でフォールバックせず、失敗を明示して記録するものとする。

## Requirement 6: チャット通知と対話照会

**User Story:** ライン監視者として、異常をチャットで即受け取り、その場で追加のログも質問したい。ツールを切り替えず状況把握するため。

#### Acceptance Criteria
1. WHEN 原因推定が完了したとき、本システムはダッシュボードのチャットに異常内容・推定原因・確信度・根拠を通知するものとする。
2. WHEN ユーザーがチャットで他ログ（温度・電流・履歴等）を尋ねたとき、本システムは該当ログを照会し回答するものとする。
3. 本システムは、通知と対話を同一のチャット面で扱うものとする。
4. IF ユーザーの照会が対象データ範囲外の場合、THEN 本システムはデータが無い旨を明示するものとする。

## Requirement 7: ダッシュボード監視

**User Story:** ライン監視者として、映像と振動を1画面で同時に監視したい。相関を目で確認するため。

#### Acceptance Criteria
1. 本システムは、検知オーバレイ付き映像とIoT時系列を同一ダッシュボードに表示するものとする。
2. WHEN 異常イベントが発生したとき、本システムはダッシュボード上で視覚的に強調表示するものとする。
3. 本システムは、直近の異常イベント一覧を参照可能に表示するものとする。

## Requirement 8: HITL 確認・修正と正解蓄積

**User Story:** 品質担当として、AIの推定が正しいか判定し、誤りを訂正したい。現場知見を記録に残すため。

#### Acceptance Criteria
1. 本システムは、各異常イベントに対しユーザーが「正しい／誤り」を判定できるものとする。
2. WHEN ユーザーが「誤り」を選んだとき、本システムは正しい原因の入力を受け付けるものとする。
3. 本システムは、AI推定・人手判定・正しい原因・時刻を1レコードとして蓄積するものとする。
4. 本システムは、蓄積レコードを後続の学習・評価に利用可能な形で永続化するものとする。

## Requirement 9: 学習ループ（few-shot 再利用）

**User Story:** 運用者として、訂正の積み重ねが推定精度に反映されてほしい。同じ誤りを繰り返さないため。

#### Acceptance Criteria
1. WHEN 新たな異常の原因推定を行うとき、本システムは過去の確定事例を検索し few-shot 文脈に含めるものとする。
2. 本システムは、人手訂正済み事例を優先的に検索対象とするものとする。
3. 本システムは、推定の正誤率などの品質指標をダッシュボードで参照可能にするものとする。

## Requirement 10: 非機能要件（稼働・コスト・セキュリティ・可観測性）

**User Story:** 審査・運用者として、動作する稼働URLを安全かつ低コストで維持したい。提出要件とガードレールを満たすため。

#### Acceptance Criteria
1. 本システムは、公開された稼働URLとして到達可能であるものとする。
2. 本システムは、待機時にスケールゼロ（min-instances=0）で運用しコストを抑えるものとする。
3. 本システムは、APIキー・シークレットをリポジトリに含めず、Vertex 呼び出しは ADC を用いるものとする。
4. 本システムは、金額・判定・破壊操作に相当する重い意思決定を決定論ロジック＋監査ログ側に置くものとする（異常確定はCVで行う）。
5. 本システムは、主要処理のログ・メトリクスを可観測基盤に出力するものとする。
6. 本システムのモデル呼び出しリージョンは us-central1、実行基盤リージョンは asia-northeast1 とする。
